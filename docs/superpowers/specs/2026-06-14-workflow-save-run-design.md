# skillsweep — workflow save/run 설계

> 작성: 2026-06-14 · brainstorming 세션 승인본 · 대상=개인 전용 도구(데이브 본인)

## 1. 배경 / 왜 이것만 짓나

외부 분석이 skillsweep를 "스킬 라우터 + 워크플로우 빌더 + 스킬 매니저"로 키우자(v0.1→v0.5)고 제안. 코드 대조 결과 두 가지가 갈렸다:

- **벽 ①(기술)** — "중복 스킬 끄기"는 구조적으로 막힘: 충돌 8군데가 전부 플러그인(agent-skills가 8/8)을 끼고 있는데, 플러그인은 **통째로만** 토글된다(낱개 끄기 미지원). 비플러그인 83개는 토글 스위치 자체가 없음(폴더 이동만). → "끄는 매니저"는 개인용이어도 반쪽. **이 설계는 끄기를 짓지 않는다.**
- **벽 ②(수요)** — 개인 전용 도구로 한정하므로 "외부 수요 n=1"은 성립 안 함(내가 곧 사용자). 제외.

두 벽을 통과하고 **벽 없이 깔끔하게 지을 수 있는 유일한 새 조각** = **워크플로우 저장/실행**. (`recommend`·`workflow 템플릿`은 이미 구현돼 있고, `workflow list`도 존재. 진짜 빠진 건 `save`·`run`.)

기대치(정직): 이 기능 자체는 수수하다. 워크플로우 모드는 "이 단계엔 이 스킬 써"라고 *안내*만 하고 실제 일은 호스트가 한다. "저장"은 그 안내 묶음을 이름 붙여 재사용 + 중복 단계의 선택을 한 번 고정해 매번 안 고르게 하는 것.

## 2. 핵심 결정 — 읽기 전용 경계의 확장

지금까지 skillsweep 정체성 = **"읽기 전용 — 아무것도 안 건드림"**. "저장"은 본질적으로 파일에 쓴다. 경계를 명확히:

> **검사·추천·실행 안내는 그대로 읽기 전용. 쓰기는 오직 "내가 저장한 워크플로우 파일" 하나에만.**
> `settings.json`·스킬 폴더·다른 스킬은 절대 안 건드림(= 벽 ①의 "끄기"는 여전히 안 함).

이 경계가 이 기능의 안전 계약이다.

## 3. 아키텍처

현재 분업(scan.mjs = 기계, SKILL.md = LLM 판단)을 그대로 확장:

- **scan.mjs** — 워크플로우 데이터 연산(save/list/delete 읽고 쓰기, 고정 스킬 실존 확인)을 담당. 순수 데이터 I/O.
- **SKILL.md** — `run`(단계별 안내)의 절차를 담당. LLM이 저장본을 읽어 대화로 walk-through. 실제 실행 없음(조언자).

## 4. 데이터 모델

워크플로우 1건:

```
{
  "name": "my-app-dev",
  "label": "내 앱 개발 흐름",
  "steps": [
    { "capability": "brainstorm", "skill": "superpowers:brainstorming", "note": "요구 정리" },
    { "capability": "tdd",        "skill": "agent-skills:test-driven-development", "note": "" },
    { "capability": "implement",  "skill": null, "note": "기본 Claude" }
  ]
}
```

- `skill` = `"출처:이름"` (중복 중 고른 그것 = 중복 해소) 또는 `null`(그때 고르기/기본 Claude).
- `capability` = scan의 cap(tdd·review·plan·debug·brainstorm·spec·ship·security·simplify) 또는 그 외(implement 등 → "기본 Claude").

## 5. 저장 위치

- **내장 5개 템플릿** = 현재 `workflows.json`(스킬 폴더, 읽기 전용·레포 추적) 그대로.
- **내가 저장한 것** = 스킬 폴더 **밖**의 사용자 파일 `~/.claude/skillsweep-workflows.json`. 재설치(폴더 덮어쓰기/clean install)해도 안 날아감. 없으면 자동 생성.
- `list`는 둘을 합쳐 "내장 / 내 것"으로 구분 표시.

## 6. 명령 (SKILL.md 라우팅 + scan.mjs 플래그)

- `workflow save <이름>` — 두 출발점:
  - (a) `recommend`/템플릿 결과에서 저장 — 중복 단계는 그 자리서 하나 골라 고정.
  - (b) 손으로 — 단계(capability) 나열 + 각 단계 스킬 고르기.
  - **입력 흐름**: scan.mjs는 비대화형(현재처럼 한 방 실행) 유지. 단계·고정 스킬을 *모으는* 건 LLM(SKILL.md)이 대화로 하고, 완성된 워크플로우 JSON을 scan.mjs에 넘겨 **저장만** 시킴(예: `--save <이름>` + JSON을 stdin/인자로). 즉 scan.mjs는 "받은 워크플로우를 사용자 파일에 쓴다"만 함.
  - **이름 충돌**: 내장 5개 템플릿 이름은 예약 — 같은 이름으로 저장하려 하면 다른 이름을 권한다(내장을 덮어쓰지 않음). 내가 저장한 것끼리 같은 이름이면 덮어쓸지 확인.
- `workflow list` — 내장 + 내가 저장한 것(출처 표시).
- `workflow run <이름>` (= `workflow <이름>`) — LLM이 단계별 안내: 각 단계의 고정 스킬을 보여주고 "이번엔 다른 걸로?" 물음. 조언만.
- `workflow delete <이름>` — 사용자 파일에서만 삭제(확인 후). 내장 템플릿은 삭제 불가(안내).

## 7. 실행 중 바꾸기(override)

고정 스킬이 기본값. "이번엔 다른 거" → 그 단계 capability의 중복 스킬들(scan 인벤토리에서)을 보여주고 **이번 실행만** 교체. 저장본은 사용자가 "이걸로 바꿔 저장"이라 해야 바뀜.

## 8. 에러 처리

- 고정해둔 스킬이 나중에 사라짐(삭제/개명) → `run`/`list`에서 "이 단계에 고정한 X가 안 보여요. 대체 후보: [지금 그 capability 하는 스킬들]" → 다시 고르기. 절대 크래시 없음.
- 사용자 파일 손상(JSON 깨짐) → 안전하게 "저장 파일을 못 읽었어요" 안내 후 빈 목록처럼 동작(내장 템플릿은 정상). 사용자 파일을 임의로 덮어쓰지 않음.
- 이름 규칙: 파일시스템/충돌 안전한 슬러그(영숫자·하이픈·밑줄·한글 허용, 경로 문자 금지).

## 9. 테스트

검사·추천 경로는 무변경(기존 손검증 유지). **새로 생긴 "쓰기"에만** 작은 `node:test`:
- save→list→run-load→delete 왕복(사용자 파일이 정확히 생기고/읽히고/지워지나).
- 같은 이름 덮어쓰기.
- 고정 스킬 실종 시 대체 후보 안내(크래시 없음).
- 손상 JSON 시 안전 동작.
- 사용자 파일은 임시 디렉터리로 격리해 테스트(실제 `~/.claude` 안 건드림).
- skillsweep 첫 테스트 — 파일을 *쓰는* 부분이라 안전망이 필요(읽기 전용 스캔과 달리).

## 10. 범위 밖 (YAGNI — 의식적 제외)

- ❌ 스킬 켜기/끄기·비활성화·백업/복구 (벽 ① 기술 한계 — 짓지 않음).
- ❌ 작업자별 프로필·팀 배정(`assign`) — 개인 도구라 불필요(벽 ② 영역).
- ❌ 워크플로우 "실행"(스킬 자동 호출) — skillsweep는 조언자, 실제 일은 호스트.
- ❌ settings.json 등 시스템 설정 쓰기 — 경계 위반.

## 11. 성공 기준

- `workflow save`로 추천 결과 또는 손작성 흐름을 이름 붙여 저장 → 재설치 후에도 `list`/`run`에 남아 있다.
- `run <이름>`이 고정 스킬로 단계 안내하고, 중복 단계에서 "이번엔 다른 거" 교체가 된다.
- 쓰기는 사용자 워크플로우 파일에만 일어난다(다른 파일 해시 불변 — 테스트로 확인).
- 검사·추천 모드 동작 무변경.
