# /skills-manager UI 접근성 — 터미널 출력 재구성 (A+C)

> 발산(7 렌즈)→수렴 세션 산출물. 확정 방향: **A(출력 재구성, 척추) + C(초심자·에러 받기, 얇은 층)**, B(대화형 입구) 보류.
> 다음 단계 진입 시 brainstorming→writing-plans→TDD로 이어받기 위한 합의 기록.

## Problem Statement
**HMW**: 비개발자가 터미널 텍스트만으로 — 색·이모지·영어 스킬명 없이 — *뭘 칠지 헤매지 않고, 결과를 한눈에 읽고, 곧장 다음 행동으로 넘어가게* 만들려면?

- 대상: 본인 + 공개 초심자(GitHub에서 처음 받는 사람).
- 성공: 첫 화면이 "읽을거리 벽"이 아니라 "결정→행동 지도"로 읽힌다.
- 제약(SKILL.md): 이모지·아이콘 금지 / 사용자 출력에 영어 스킬·함수 이름 금지(흔한 개발 용어 TDD·디버깅·스펙·리뷰는 OK) / 내부 전문어 금지(숫자는 디지털 그대로) / 담백 / 한 줄 결론 먼저.

## Recommended Direction
**A(출력 재구성)가 척추다.** 사용자가 고른 UI 범위가 "출력 가독성"이고, 7개 렌즈 중 4개가 독립적으로 같은 두 곳에 모였다 — ① 맨 끝 고정 `쓰는 법:` 한 줄을 *네 결과에서 나온* '다음 한 수' 블록으로, ② 28줄 벽을 결론+큰 겹침 몇 개로 접고 상세는 `--all`로. 이 둘은 한 몸이다: '다음 한 수'는 줄을 더하고 '접기'는 더 많이 빼서, 합치면 화면이 순감한다. 엔진(스캔·판정)은 손대지 않고 `scan.mjs` 출력부(317~342줄)만 다시 짜므로 위험이 낮다.

**C(초심자·에러 받기)를 얇게 얹는다.** "공개 사용자도"라는 답 때문에 살아나는 층 — 첫 실행 환영 4줄(일회성, '읽기 전용=안전'을 *사실로* 못박되 유치체 금지)과, 가장 불친절한 지점인 저장/수정 에러를 관대하게(빈 입력·잘못된 단계번호에 사용법+단계목록 회신). 단골에겐 안 보이므로 비용이 누적되지 않는다.

**B(도구가 먼저 묻기)는 의도적으로 뺀다.** '담백·조언자' 철학과 마찰하고, 매번 메뉴가 뜨면 단골에 잔소리가 되며, 사용자 1순위(가독성)에 빗나간다. 그 가치는 A의 '다음 한 수' 블록이 약화판으로 대부분 흡수한다.

## Key Assumptions to Validate
- [ ] **상세를 `--all`로 미뤄도 안 답답하다** → 본인이 일주일간 접힌 기본 출력만으로 써보기(도그푸딩).
- [ ] **'다음 한 수'가 실제로 행동을 유발한다** → 바뀐 출력 본 뒤 recommend/save를 실제로 쓰는지 관찰.
- [ ] **첫 실행 환영이 '경고 벽' 아닌 '내가 받은 지도'로 읽힌다** → 도구 처음 보는 비개발자 1명에게 보여주고 첫 반응 확인.
- [ ] **복붙 저장 줄이 Windows에서 안 깨진다** → PowerShell에서 `echo JSON | --save` 실측(깨지면 대화형 경로만).

## MVP Scope
**들어가는 것 (A — 출력 재구성):**
1. 기본 출력 접기 — 제목+한 줄 결론(항상; 현재는 conflicts 있을 때만)을 빈 줄로 격리 / 큰 겹침 상위 N개만 미리보기 / '깔린 스킬 묶음별 카운트'와 '끄기' 장문은 `--all`로, 기본엔 "끌 건 없음" 5글자만.
2. 맨 끝 `쓰는 법:` → 결과 기반 '다음 한 수' 3줄(`covSorted[0]`·`conflicts[0]` 활용, 각 항목에 '왜 누르나' 한 줄). `--judge`/`--json`엔 안 붙임.
3. 숫자 열 맞춤 — 겹침 미리보기·workflow list 단계표를 기존 `padW`/`dispWidth`로 정렬.
4. 새 플래그 `--all`(=기존 전체 벽). `--judge`는 판정 패킷 용도 그대로(과적재 금지).

**들어가는 것 (C — 초심자·에러):**
5. 첫 실행 환영(일회성) — 저장 파일 없음(`loadUser().length===0`) + 인자 없는 기본 호출일 때만 분석 위 4줄, 둘째 실행부터 생략.
6. 관대한 에러 — `--save` 빈 stdin / `--set-skill` 잘못된 단계번호에 단계 목록 동반(`workflow-store` bad-step 반환에 라벨 추가 = 작은 계약 변경).
7. recommend→save 잇기 — 결과 끝 "이대로 저장하려면" 안내(복붙형은 실측 후, 우선은 "'이걸로 저장'이라 말하면 호스트가 이음" 대화형).

## Not Doing (and Why)
- **B / 진입에 번호 메뉴·먼저 묻기** — 담백·조언자 철학 마찰 + 단골 잔소리 + 가독성에 빗나감. A의 '다음 한 수'로 흡수.
- **색·이모지·아이콘** — SKILL.md 절대 제약.
- **HTML·새 표면** — 사용자가 "출력 가독성"을 골랐고 "새 표면"은 안 골랐음(project-doctor HTML 렌더러 3회 기각과 같은 철학).
- **스킬 끄기** — 플러그인 통째 토글 구조적 벽, 영구 미구현. 범위 밖.
- **정렬을 2단 '진짜 N곳'으로 자동 재정렬** — 1단 scan은 LLM 판정을 모름. 자동화하면 역할-다른 가짜 겹침이 위로 올라가 오정렬. 정렬은 1단 수치로만, 보정은 LLM 단계(SKILL.md)에 위임.

## Open Questions
- 큰 겹침 미리보기 N = 5? 7? "8개 이하면 안 접음" 경계를 둘까?
- 상세 플래그 이름 `--all` vs `--full` — `--judge`와 역할 분리 확정.
- 정렬 표기: 1단 수치로 정렬하되 "N곳(넓게)" 라벨 유지로 충분한가, 아니면 정렬 자체 보류?
